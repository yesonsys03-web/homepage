# Page Edit 개발 태스크 분해본

> 기준 문서: `docs/page_edit.md`
>
> 목적: 구현 전에 작업 단위를 명확히 쪼개고, 우선순위/의존성/완료 기준을 고정한다.
>
> 범위: 설계/개발/테스트/배포 준비. (실제 구현 제외)

## 0) 작업 원칙

- 한 태스크는 한 가지 결과물을 만든다.
- 모든 태스크는 완료 기준(DoD)을 가진다.
- 권한/감사/롤백은 "기능"이 아니라 "필수 제약"으로 취급한다.

## 1) Epic 구성

- Epic A: 데이터 모델 + API 기반
- Epic B: 편집기 UX(블록/프리뷰)
- Epic C: Draft/Publish/Version 운영 흐름
- Epic D: 품질(검증/테스트/관측)
- Epic E: 릴리즈/전환

## 1.1) 사전 확인 결과 (코드베이스 팩트)

- `/admin/pages` 라우트는 이미 존재한다 (`src/App.tsx`).
- Admin action log 인프라는 이미 존재한다 (`server/db.py`의 `admin_action_logs`, `server/main.py`의 `write_admin_action_log`, `/api/admin/action-logs`).
- 따라서 본 문서의 핵심은 "신규 인프라 존재 여부"가 아니라 "페이지 편집 버전/권한/운영 흐름 정합성"에 있다.

## Epic A — 데이터 모델 + API 기반

### A-1. 페이지 문서 스키마 확정

- 설명: `PageDocument` 구조를 서버/클라이언트 공통 계약으로 확정
- 산출물:
  - 스키마 명세서(필드, 타입, 제약)
  - 블록 타입별 `content` 스펙
- 의존성: 없음
- DoD:
  - 필수 필드/옵셔널 필드가 문서화됨
  - version 증가 규칙이 정의됨

### A-2. 버전 저장 모델 설계

- 설명: draft/published + version snapshot 저장 구조 설계
- 산출물:
  - 테이블/컬렉션 모델 초안
  - rollback 시점의 참조 규칙
- 의존성: A-1
- DoD:
  - rollback 대상 식별 키가 확정됨
  - 최신본/히스토리의 정합성 규칙이 문서화됨

### A-3. API 계약 확정

- 설명: `draft 조회/저장`, `publish`, `versions`, `rollback` API 스펙 확정
- 산출물:
  - 엔드포인트별 request/response 스펙
  - 에러 코드/권한 오류 스펙
- 의존성: A-1, A-2
- DoD:
  - 프론트/백엔드가 같은 계약 문서를 기준으로 개발 가능

### A-4. 권한 정책 상세화

- 설명: admin/super_admin 권한 행위표(Action Matrix) 정의
- 산출물:
  - 액션별 허용 역할 표
  - reason 필수 액션 목록
- 의존성: A-3
- DoD:
  - publish/rollback 권한이 모호함 없이 정의됨

---

## Epic B — 편집기 UX(블록/프리뷰)

### B-1. 편집기 정보 구조(IA) 고정

- 설명: `/admin/pages` 하위 탭 구조와 상태 전환 정의
- 산출물:
  - IA 다이어그램
  - 탭 간 상태 보존 규칙
- 의존성: A-3
- DoD:
  - 화면 이동 시 draft state 처리 규칙이 확정됨

### B-2. 블록 에디터 MVP 스펙

- 설명: Hero/RichText/Image/CTA 4개 블록 우선 스펙화
- 산출물:
  - 블록별 편집 필드 정의
  - 블록 공통 액션(추가/복제/삭제/정렬)
- 의존성: B-1
- DoD:
  - 블록별 최소/최대 제약이 문서화됨

### B-3. 프리뷰 모드 설계

- 설명: desktop/mobile 프리뷰 토글과 렌더 규칙 정의
- 산출물:
  - 프리뷰 상태 모델
  - 렌더 fallback 규칙
- 의존성: B-2
- DoD:
  - 동일 draft를 두 디바이스에서 일관 렌더 가능하도록 규칙 확정

### B-4. 편집 UX 가드레일

- 설명: 저장 전 검증 체크리스트 정책 정의
- 산출물:
  - 필수값/길이/이미지 비율/금칙어 검증 규칙
  - 차단(blocking) vs 경고(warning) 분류
- 의존성: B-2
- DoD:
  - 게시 가능 여부 판단 기준이 자동화 가능 수준으로 명시됨

---

## Epic C — Draft/Publish/Version 운영 흐름

### C-1. Draft 저장 플로우 정의

- 설명: 자동 저장/수동 저장 정책, 충돌 처리 정책 수립
- 산출물:
  - 저장 정책 문서
  - 충돌 발생 시 UX 플로우
- 의존성: A-3, B-1
- DoD:
  - 데이터 유실 방지 시나리오가 정의됨

### C-1a. 충돌 감지 전략 분리 설계 (MVP 범위 결정)

- 설명: 다중 편집 충돌 처리 전략을 별도 태스크로 분리
- 산출물:
  - 전략안 A: 낙관적 동시성(버전 비교 + 저장 거부)
  - 전략안 B: 단기 편집 락(만료/강제 해제 포함)
  - MVP 포함/제외 결정 기록
- 의존성: C-1, A-2
- DoD:
  - 전략 1개가 최종 채택되고 이유가 문서화됨
  - MVP 제외 시 후속 스프린트 배치가 명시됨

### C-2. Publish 플로우 정의

- 설명: publish 전 확인, 권한 확인, 감사 로그 필수화
- 산출물:
  - publish 체크리스트
  - reason 입력 UX 규칙
- 의존성: C-1, A-4
- DoD:
  - publish 실패 케이스와 복구 경로가 명시됨

### C-3. Version 비교/롤백 플로우

- 설명: 버전 리스트, diff, rollback UX 및 정책 정의
- 산출물:
  - diff 표시 범위 규칙
  - rollback 승인 규칙
- 의존성: A-2, C-2
- DoD:
  - 롤백 후 검증 단계가 문서화됨

---

## Epic D — 품질(검증/테스트/관측)

### D-1. 테스트 전략 수립

- 설명: 단위/통합/E2E 테스트 범위와 우선순위 정의
- 산출물:
  - 테스트 매트릭스
  - 회귀 우선순위 목록
- 의존성: A~C
- DoD:
  - 필수 회귀 케이스가 누락 없이 나열됨

### D-2. 감사 로그/관측 포인트 정의

- 설명: 어떤 액션을 어떤 필드로 로깅할지 표준화
- 산출물:
  - 이벤트 스키마(예: draft_saved, page_published, page_rolled_back)
  - 운영 대시보드 지표 목록
- 의존성: C-2, C-3
- DoD:
  - 보안/감사 관점 추적 가능성 확보

### D-3. 성능 기준선 설정

- 설명: 편집기 초기 로딩/저장/프리뷰 렌더 성능 기준 정의
- 산출물:
  - SLO 초안
  - 측정 방식
  - 측정 도구 확정(예: Lighthouse CI, Web Vitals, 커스텀 편집 이벤트 메트릭)
- 의존성: B-3
- DoD:
  - 성능 회귀 판단 기준이 정량화됨
  - CI/스테이징에서 동일 도구로 재현 측정 가능

---

## Epic E — 릴리즈/전환

### E-1. 단계적 롤아웃 계획

- 설명: 내부 사용자부터 점진 노출 전략 수립
- 산출물:
  - 롤아웃 단계표
  - 롤백 트리거 조건
- 의존성: D-1, D-2
- DoD:
  - 실패 시 즉시 이전 체계 복귀 절차가 정의됨

### E-2. 데이터 마이그레이션 계획

- 설명: 기존 페이지 데이터를 블록 구조로 이행하는 전략 수립
- 산출물:
  - 마이그레이션 규칙표
  - 샘플 변환 결과
- 의존성: A-1, A-2
- DoD:
  - 핵심 페이지 변환 정확도 기준 합의

### E-3. 운영 가이드 문서화

- 설명: 편집자/승인자 운영 매뉴얼 작성
- 산출물:
  - 운영 핸드북
  - 장애 대응 FAQ
- 의존성: C 전반, E-1
- DoD:
  - 신규 운영자 온보딩 가능 수준의 문서 완성

2. A-2 버전 저장 모델
3. A-3 API 계약 확정
4. A-4 권한 정책 상세화
5. B-1 편집기 정보 구조(IA) 고정
6. C-1 Draft 저장 플로우
7. C-2 Publish 플로우
8. B-2 블록 에디터 MVP 스펙
9. B-3 프리뷰 모드 설계
10. B-4 편집 UX 가드레일
11. C-1a 충돌 감지 전략 분리 설계
12. C-3 Version/롤백 플로우
13. D-1 테스트 전략
14. E-1 단계적 롤아웃

## 3) 스프린트 제안 (2주 단위)

### Sprint 1

- A-1, A-2, A-3, A-4, B-1
- 게이트: 설계 리뷰 통과

### Sprint 2

- C-1, C-2, B-2, B-3, B-4, C-1a
- 게이트: 내부 QA 시나리오 통과

### Sprint 3

- C-3, D-1, D-2, D-3, E-1, E-2, E-3
- 게이트: 스테이징 운영 점검 완료

### Sprint 4 (선택)

- 예약 발행, 승인 코멘트 고도화, 확장 블록 타입
- 게이트: 운영팀 파일럿 만족도 확인

## 4) 확인 질문(개발 전 합의 필요)

- publish 권한을 super_admin 100% 고정할지, 승인 워크플로우를 둘지
- 기존 페이지 데이터 변환 범위를 전량/핵심 페이지로 나눌지

## 4.1) 범위 확정 메모 (권장 기본값)

- 예약 발행: MVP 제외, Sprint 4로 이관
- 충돌 감지: MVP에서는 "낙관적 동시성" 우선 검토, 락 모델은 후속
- 블록 diff: MVP는 텍스트/핵심 필드 diff, 구조 diff는 후속

## 5) 정의된 완료(Definition of Done)

- 기능 DoD: 요구사항/권한/실패 시나리오 문서 완료
- 품질 DoD: 테스트 매트릭스에 필수 케이스 매핑 완료
- 운영 DoD: 롤백 절차 + 로그 추적 항목 문서화 완료
